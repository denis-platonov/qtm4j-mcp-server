# Release 1.1.0 Design

## Context

The repository is ahead of the published npm package.

- `package.json` is `1.1.0`
- `server.json` is `1.0.0`
- npm latest for `@denis-platonov/qtm4j-mcp-server` is `1.0.0`

The existing GitHub Actions release workflow publishes on tags matching `v*` and explicitly requires the tag version to match both `package.json` and `server.json`.

## Goal

Release version `1.1.0` so npm and MCP Registry metadata reflect the current repository state.

## Release Notes Scope

The release notes should call out the functional delta from `v1.0.0`, not just the manifest alignment needed to publish.

Recommended highlights for `1.1.0`:

- Expanded test case search with paginated `search_test_cases`
- Bulk listing with `list_all_project_test_cases`
- Richer test case creation with folder placement and auto-pick folder support
- New folder management tools:
  - `list_test_case_folders`
  - `create_test_case_folder`
  - `add_test_case_to_folders`
  - `remove_test_case_from_folders`
- New test case inspection tools:
  - `get_test_case`
  - `get_test_case_details`
- New step management tools:
  - `get_test_case_steps`
  - `add_test_case_steps`
  - `update_test_case_step`
- Test case metadata editing:
  - `update_test_case_summary`
  - `update_test_case_description`
- Better client compatibility for numeric inputs passed as strings, especially for `projectId` and pagination parameters
- Documentation updates for local Cursor usage so users can verify they are running the newer local build when npm is behind

README adjustments needed as part of the release:

- Expand the `## Tools` table so it matches the current 21 registered tools instead of the older 8-tool subset
- Update the publish instructions to prefer the tag-driven GitHub Actions release flow over manual `npm publish`
- Update the release example from `v1.0.0` to `v1.1.0`
- Keep the local-build guidance because it explains why npm and the current branch may differ before the new release is published

## Draft Release Notes

### Added

- `search_test_cases` for paginated raw QTM4J test case search
- `list_all_project_test_cases` for multi-page project-wide test case retrieval
- Folder management tools for listing, creating, adding, and removing test case folders
- Test case detail helpers for fetching current and version-specific records
- Test step helpers for listing, creating, and updating steps
- Test case summary and description update tools

### Changed

- Enhanced `create_test_case` with folder placement and auto-folder selection options
- Improved parameter coercion so MCP clients that send numbers as strings work more reliably
- Expanded README guidance for local Cursor configuration and tool verification

## Recommended Approach

Use the existing tag-driven release workflow.

1. Update `server.json` from `1.0.0` to `1.1.0` so it matches `package.json`.
2. Run local verification with `npm run build` and `npm run test:run`.
3. Commit the release-prep change on the current branch.
4. Create tag `v1.1.0`.
5. Push the branch and tag to GitHub.
6. Let the existing `Release` GitHub Actions workflow publish the npm package and MCP Registry metadata.

## Alternatives Considered

### Revert back to 1.0.0

This would make the manifests internally consistent, but it would abandon unreleased work that is already present in the repository.

### Publish manually with npm

This would bypass the repository's existing release validation and increase the chance of npm and MCP Registry metadata drifting again.

## Components Affected

- `server.json`
- `README.md`
- release notes / GitHub release description
- Git tag `v1.1.0`
- GitHub Actions `Release` workflow
- npm package `@denis-platonov/qtm4j-mcp-server`
- MCP Registry metadata for `io.github.denis-platonov/qtm4j`

## Data Flow

1. Local repo versions are aligned.
2. Local build and test verification passes.
3. Git tag `v1.1.0` is pushed.
4. GitHub Actions validates the tag against `package.json` and `server.json`.
5. Workflow publishes to npm.
6. Workflow publishes `server.json` metadata to the MCP Registry.

## Error Handling

- If local build or tests fail, stop before tagging.
- If tag validation fails in CI, fix manifest mismatch and retag with the correct version.
- If npm publish fails, inspect repository secrets and package ownership.
- If MCP Registry publish fails, inspect workflow logs and registry authentication step output.

## Testing

Required local verification before tagging:

- `npm run build`
- `npm run test:run`
- Quick README review to confirm the tool list and release instructions match the shipped version

Optional post-release verification:

- Confirm npm latest is `1.1.0`
- Confirm GitHub release workflow completed successfully
- Confirm registry metadata reflects `1.1.0`

## Scope

This release design covers aligning version metadata, updating README documentation, shipping `1.1.0`, and describing the already-landed tool additions accurately in release notes. It does not include new feature implementation or workflow redesign.
